# 检索"还是差"的根因诊断与研究方案

> 基准跑：`target/retrieval-regression/live_embedding-full_live-1780583714/report.json`
> 结论先行：**当前看到的"差"约一半是测试集 ground-truth 设计缺陷造成的伪失败，不是算法失败。** 真正的算法残留问题只有两类，且有成熟论文方法对症。

---

## 一、把 88 道回答题彻底分解（不是 gating 问题）

| 桶 | 数量 | 含义 |
|---|---|---|
| 答了 & 文档命中 | 18 | 真正确 |
| 答了但目标文档不在 top3 | 26 | **docRank 全=None**：目标文档根本没进候选，却用别的文档答了 |
| 文档在 top3 却被 gate 拒 | 34 | gating 误拒（真问题，但次要） |
| 真召回失败（文档 rank 4–23） | 10 | 真召回弱 |

→ 目标文档进不了 top3 的有 **36/88 = 41%**，远超 gating 误拒。**主矛盾在文档召回/排序，不在 gating。** 上一轮 gating 微调把整体从 0.51 抬到 0.55 就到顶了，因为天花板在召回侧。

---

## 二、根因：测试语料的 ground-truth 是病态的（决定性发现）

读 `Memory_Test/generate_test_docs.py::build_body`（:89-131）：每个主题（20 个）生成 5 个文档（制度/会议纪要/SOP/复盘/问答卡），**5 个文档都嵌入了一段逐字相同的 `## 唯一事实卡`**（含 核心事实 / 指标口径 / 负责人 / 反常识提醒）。5 个兄弟文档之间只有顶部一节标题不同（制度=适用范围与强制边界，会议纪要=决策记录与待办…）。

后果——量化：
- **88 道回答题里 44 道（50%）是"共享事实卡"查询**（"{代号}的唯一事实卡里核心事实是什么"类），其答案在同主题 5 个文档里**字节级相同**。
- 但测试集给每道题只标了 **1 个**目标文档（44 道里 38 道标的是"制度"文档，纯属武断）。
- 这 44 道里 **23 道被判 `top3_document_recall=False`**；而其中 **17 道系统实际产出了正确答案**（在某个兄弟文档里找到了同一段事实），chunk 召回也只认了 6/23——因为"目标 chunk"被绑死在那个武断选中的文档上。

**即：约 23 道"召回失败"是评测假阳性。系统检索到了含有正确答案的兄弟文档，却因为"文档 ID 对不上"被扣分。** 当前 top3_doc_recall≈0.59 里掺了大量这种噪声，盯着它调算法等于对着噪声优化。

> 这是 IR 评测的经典陷阱：**relevance judgments 不完整 / 文档重复**（TREC 早有讨论）。对 RAG 应当用 **nugget / answer-containing** 口径（答案事实是否出现在被引证据里），而非文档 ID 精确匹配（参 RAGAS 的 context-recall、TREC nugget 评测）。

---

## 三、剥掉伪失败后，真正的算法问题只剩两类

### 真问题 A：同代号兄弟文档的细粒度区分（section-specific 查询，44 道）
另外 44 道是"段落特定"查询（问会议纪要的待办、复盘的偏差原因、SOP 的执行步骤…），**答案只在某一个 doc-type 里**。这才是真检索难点：query 命中代号→召回整簇 5 个兄弟，但要排出正确的那一个。当前 dense 向量对 5 个高度相似兄弟给出近乎相同的分，RRF/rerank 难以稳定区分，目标常落到 rank 4–6。

**对症论文 / 方法：**
1. **Contextual Retrieval（Anthropic, 2024）** — 索引时给每个 chunk 前置一段"它在文档中的语境"（doc-type + 代号 + 章节），再做 embedding 和 BM25。官方报告检索失败率 −49%（叠加 rerank −67%）。**直接解决"同代号兄弟无法区分"**：让"会议纪要"的 chunk 嵌入里带上"这是 银杏-17 的会议纪要"。
   - 本语料结构极规整（文件名+H1 已含 类别/项目/代号/章节），可用**确定性模板前缀**（如 `技术架构 / 银杏-17 / ARC-17 / 会议纪要`）拼进 chunk 的 embedding 文本与 FTS 文本，**零 LLM 成本、可复现**，是 Contextual Retrieval 的廉价特例。
2. **doc-type 查询路由（faceted / metadata filtering）** — 从 query 检出目标文档类型意图（"会议纪要/复盘/SOP/问答卡/制度/待办/偏差/步骤"等触发词）→ 对应 doc-type 的兄弟加权。经典且便宜。
3. **Proposition / Dense-X（Chen et al., 2023）** — 把"唯一事实卡"按命题原子化索引（适合共享事实卡类，但对区分兄弟帮助有限，优先级低）。
4. **Late interaction / ColBERTv2** — token 级细粒度匹配，对兄弟区分最强，但工程重，本地化成本高，**暂不建议**。
5. **HyDE（Gao et al., 2022）** — 生成假设答案再检索，缓解"改写式回忆"词表不匹配；与本系统主打场景契合，可作 B 阶段实验。

### 真问题 B：gating 误拒（34 道，次要）
已在 `docs/qa/gating_recall_fix_plan.md` 给出方案（coverage 少词全覆盖档 + 实体已落地放行）。**安全垫已验证**：11 道正确拒答里 10 道靠 `entity_not_grounded`/`intent_blocked` 硬拦守住，抬软 gate 分不会放跑它们。等评测口径修正后再回归这一项，避免对噪声调参。

## 四、行动方案（按"先让指标可信，再优化算法"排序）

### P-A（最高优先：修评测，否则无法度量任何算法收益）
改回归 harness（`memori-core/examples/retrieval_regression.rs` + 评分逻辑），把文档级 ground-truth 从"单文档精确匹配"改为**二选一或并用**：
- **(a) 主题级 relevance**：同主题任一兄弟文档命中即算 doc-recall 命中（在 suite 里给每题加 `acceptable_documents` = 同主题同事实卡的兄弟集合）；
- **(b) nugget answer-recall**：被引证据文本是否包含 `target_clues`（核心事实串）即算命中，弱化文档 ID。
- 建议先做 (a)，改动小、可立刻重判历史跑；(b) 作为更贴近 RAG 真实价值的二级指标并行展示。
- **预期**：仅修评测口径，真实 doc-recall 会从 ≈0.59 跳到显著更高（23 道伪失败里至少 17 道转正），让后续算法改动有可信基线。

### P-B（算法核心：Contextual Retrieval 廉价版）
索引阶段给每个 chunk 前置确定性语境前缀（类别/项目/代号/章节，来自文件名+H1），同时进入 embedding 文本与 FTS 文本。预期对 44 道 section-specific 查询的兄弟区分有结构性提升。需一次全量重建索引。

### P-C（算法补强：doc-type 查询路由）
query 侧抽取 doc-type 意图词 → 命中时对应 doc-type 兄弟加召回/排序权重。与 P-B 叠加。

### P-D（gating，放最后）
评测口径修正后，按 `gating_recall_fix_plan.md` 收尾 34 道误拒。

### 验收
- 先只上 P-A，重判最新跑 → 确认真实 doc-recall 基线。
- 再上 P-B（+重建索引）跑 #N+1，对比 section-specific 子集的 docRank 改善。
- 全程用 `scripts/export-retrieval-regression-excel.py` 记录，正式对比从这条"评测口径修正后"的新基线开始画折线。

## 五、给测试集设计者的提醒（可选，治本）
若想让文档级 ground-truth 本身良定义，治本做法是让 5 个兄弟文档的"唯一事实卡"**各自不同**（每个 doc-type 只放属于自己的那部分事实），而不是 5 份逐字复制。但这会重置历史可比性；在那之前，P-A 的主题级/nugget 评测是更务实的过渡。
